# MQTT Alarm Event Bridge Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Publish a JSON alarm event to MQTT topic `jetson/alarm` every time `monitor_node` triggers alarm audio playback, using a ROS topic bridge so `monitor_node` stays MQTT-free and `mqtt_node` remains the single MQTT integration point.

**Architecture:** `monitor_node` publishes a JSON string to a new ROS topic (recommended: `alarm/event`) whenever an alarm playback is triggered. `mqtt_node` subscribes to that ROS topic and republishes the payload to MQTT topic `jetson/alarm` through the existing Paho client.

**Tech Stack:** ROS 2 Humble, rclcpp, std_msgs, JSONCPP, Paho MQTT C++, GoogleTest.

## Notes

- Keep MQTT ownership inside `mqtt_node`; do not embed Paho MQTT into `monitor_node`.
- Use `std_msgs::msg::String` for minimal change; custom message types are out of scope for this iteration.
- Prefer monitor to publish already-shaped JSON so mqtt_node only bridges, not transforms.
